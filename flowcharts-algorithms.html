Flowcharts — shapes, rules, and how to draw one
A flowchart is a picture of an algorithm. Every shape has a fixed meaning. Cambridge Paper 2 asks you to complete or draw one almost every year.
What is a flowchart? A flowchart turns your algorithm into a diagram so anyone can follow the logic without reading code. You draw boxes and arrows. Each box shape tells the reader exactly what kind of action is happening. Arrows show which box comes next — and arrows that loop back create a repetition.
The 5 shapes you must know — every single one:
OVAL → Terminal — START or END. Every flowchart has exactly one START oval at the top and one END oval at the bottom. Nothing goes before START or after END. If you forget to draw them, you lose marks.
RECTANGLE → Process — an action or assignment. Any instruction that does something goes here: count ← 0, index ← index + 1, temp ← A[j]. It has one arrow in and one arrow out.
DIAMOND → Decision — a YES / NO question. Always a condition that is either true or false: index < 20?, found = TRUE?. It always has TWO arrows leaving it — one labelled YES, one labelled NO. Never more, never fewer.
PARALLELOGRAM → Input / Output — reading in or printing out. Use this for: INPUT x, OUTPUT result, OUTPUT count. Never put a calculation inside a parallelogram — that goes in a rectangle.
→ ↓ ↑ → Flow lines — direction of execution. Arrows must always have a direction. An arrow pointing DOWN or RIGHT moves forward through the algorithm. An arrow pointing UP or LEFT creates a loop — it takes you back to repeat something. Always label YES / NO on decision arrows.
How to read a flowchart — step by step for beginners:
  • 1 Always start at the START oval. Find it — it's the only rounded oval at the very top. Follow the arrow leaving it downwards.
  • 2 When you hit a rectangle, do the action written inside it (assign the variable, do the calculation) then follow the single arrow out.
  • 3 When you hit a diamond, ask yourself: is the condition TRUE or FALSE right now? Follow the YES arrow if true, the NO arrow if false.
  • 4 When an arrow points upwards, you are going back around a loop. Follow it up to whatever box it points to and continue from there.
  • 5 When you reach the END oval, stop. The algorithm is finished.

Why count ← 0 and index ← 0 first? These are called initialisations. You must set every variable to a starting value before the loop begins. If you forget, the variable could contain garbage data and give a wrong answer.
Why is the diamond index < 30 and not index <= 30? The array has indices 0 to 29. When index reaches 30, there is no Data[30] — it doesn't exist. So the condition must be index < 30. When it becomes false, we exit the loop safely.
Why is index ← index + 1 inside the YES branch? We only move to the next element if we are still inside the loop (index < 30 is true). The increment goes after the inner check so every valid element gets examined before we advance.
The loop back arrow — this is the loop! After incrementing index, the arrow goes back UP to the first diamond (index < 30?). This creates the repetition. Without this arrow going back, the algorithm only runs once.
How a beginner should tackle any "complete the flowchart" question:
  • 1 Read the question and list: what are the initialisations? Look for variables that need a starting value (usually 0 or FALSE or –1). Draw each one as a rectangle right after START.
  • 2 Find the loop condition. Ask: what keeps the loop going? This becomes the diamond. It is almost always something like index < n or NOT found.
  • 3 Find any inner check inside the loop. If the question says "only do X when condition Y is true", that's another diamond inside the YES branch of the loop diamond.
  • 4 Add the counter or action. Whatever the question asks the algorithm to DO goes in a rectangle on the YES branch of the inner diamond.
  • 5 Increment the index at the bottom of the loop body (rectangle), then draw the arrow going back UP to the loop diamond. This is the most commonly forgotten step.
  • 6 When the loop exits (NO branch of the loop diamond), draw the output parallelogram and then the END oval.

HintDeclare: Found as BOOLEAN, Index as INTEGER. Set Found ← FALSE, Index ← -1. WHILE NOT Found AND Index < 49. Inside: Index ← Index + 1, then check if Items[Index] = Target → Found ← TRUE. After loop: IF Found THEN RETURN Index ELSE RETURN -1.
FUNCTION FindItem(Target : STRING) RETURNS INTEGER DECLARE Found : BOOLEAN DECLARE Index : INTEGER Found ← FALSE Index ← -1 WHILE NOT Found AND Index < 49 DO Index ← Index + 1 IF Items[Index] = Target THEN Found ← TRUE ENDIF ENDWHILE IF Found = TRUE THEN RETURN Index ELSE RETURN -1 ENDIF ENDFUNCTION

StudentList is a global 1D array of type STRING containing 200 elements, sorted in ascending alphabetical order. The global variable ListSize stores the number of valid entries.

Write pseudocode for a function BinarySearch() that takes a STRING parameter SearchName, performs an iterative binary search, and returns the index if found or –1 if not found. Use Found and SearchFailed flags.
  • Correct function header, parameter, and RETURNS type [1]
  • All four variables declared with correct types: Found, SearchFailed, First, Last, Middle [1]
  • Correct initialisations of all five variables [1]
  • WHILE condition uses both NOT Found AND NOT SearchFailed [1]
  • Middle calculated with DIV; all three ELSE IF branches correct [1]
  • Correct RETURN values: Middle when found, –1 when not found [1]
HintDeclare Found, SearchFailed as BOOLEAN. Declare First, Last, Middle as INTEGER. Initialise Found ← FALSE, SearchFailed ← FALSE, First ← 0, Last ← ListSize – 1. Inside WHILE: calculate Middle ← (First+Last) DIV 2. Then IF equal → Found; ELSE IF First>=Last → SearchFailed; ELSE IF middle too big → Last←Middle–1; ELSE First←Middle+1.
FUNCTION BinarySearch(SearchName : STRING) RETURNS INTEGER DECLARE Found, SearchFailed : BOOLEAN DECLARE First, Last, Middle : INTEGER Found ← FALSE SearchFailed ← FALSE First ← 0 Last ← ListSize - 1 WHILE NOT Found AND NOT SearchFailed DO Middle ← (First + Last) DIV 2 IF StudentList[Middle] = SearchName THEN Found ← TRUE ELSE IF First >= Last THEN SearchFailed ← TRUE ELSE IF StudentList[Middle] > SearchName THEN Last ← Middle - 1 ELSE First ← Middle + 1 ENDIF ENDWHILE IF Found = TRUE THEN RETURN Middle ELSE RETURN -1 ENDIF ENDFUNCTION
Q8 Evaluate — choose the right algorithm
Hard [6 marks] ▼ expand
A program stores 1000 student names in a 1D array. Names are added regularly. The array is kept sorted in ascending alphabetical order at all times. A teacher searches for a student name approximately 80 times per day.
  • (a) Explain, with reference to the number of comparisons, why binary search is more efficient than linear search for this use case. [2]
  • (b) The programmer currently uses insertion sort to maintain the sorted order when a new name is added. A colleague suggests using bubble sort instead. Give one advantage and one disadvantage of using bubble sort compared to insertion sort in this context. [2]
  • (c) State one situation where a linear search would be more appropriate than a binary search. [1]
  • (d) State what must be done to the array before a binary search can be used, and name an algorithm that achieves this. [1]
HintFor (a): linear search checks up to n elements in the worst case; binary search checks at most log₂(n). For n=1000 that is about 1000 vs 10 comparisons. For (b): both are O(n²) but insertion sort makes fewer comparisons and shifts when the list is already nearly sorted — which is the case here since the list is maintained in order. For (c): think about when the data is NOT sorted.
(a) Linear search: worst case checks all 1000 elements per search. Binary search: worst case checks only ≈ 10 elements (log₂1000 ≈ 10). With 80 searches per day, binary search requires far fewer total comparisons — making it significantly more efficient here. (b) Advantage of bubble sort: simple to understand and implement. Disadvantage: when a new name is added the list is nearly sorted — insertion sort is more efficient in this case because it stops the inner loop early when no more shifts are needed, whereas bubble sort always completes all n–1 passes. (c) Linear search is more appropriate when the array is unsorted (binary search requires a sorted array), or when the array is very small (the overhead of sorting is not worth it). (d) The array must be sorted in ascending order. Algorithm: bubble sort or insertion sort.
