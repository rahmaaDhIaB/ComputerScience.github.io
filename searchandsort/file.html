Records
Grouping related data under one name — like a row in a spreadsheet
The F1 analogy Imagine a spreadsheet of F1 teams. Each row is one team: position, team name, two drivers, points. All different data types but they belong together — that's a record. Instead of 5 separate variables floating around, you bundle them into one named thing.
Think of it like a spreadsheet row
Position TeamName Driver_1 Driver_2 Points
1 Red Bull Racing Verstappen Perez 860
2 Mercedes Hamilton Russell 409
3 Ferrari Leclerc Sainz 406
Each row above = one record instance. The column headers = the record type (the template).
Declaring a record in Python
Python has no built-in "record" keyword. We fake it with a class. But it's a SIMPLE class — no private attributes, no getters, no setters. Just a container.
# ── DECLARE THE RECORD TYPE ────────────────────────────────
class TeamRecord:
    def __init__(self, position, team_name, driver_1, driver_2, points):
        self.Position = position      # INTEGER  ← NO __ before name = PUBLIC
        self.TeamName = team_name     # STRING   ← NO __ before name = PUBLIC
        self.Driver_1 = driver_1      # STRING
        self.Driver_2 = driver_2      # STRING
        self.Points   = points        # INTEGER

Instantiating a record
Creating actual record instances with real values.
# ── CREATE RECORD INSTANCES ────────────────────────────────

team2 = TeamRecord(2, "Mercedes",       "Hamilton",  "Russell", 409)

print(team1.Points)      # output: 860

teams = [team1, team2]
for team in teams:
    print(team.Position, team.TeamName, team.Points)
Your question: "Is a record the same as an OOP class?"
YES — the Python syntax is almost identical. The only difference is one character: __
RECORD — public fields

    self.Points = pts

c.__Name — BLOCKED (AttributeError)

Summary in one sentence: A record and an OOP class are written the same way in Python. The only difference is that records use self.FieldName (public) and OOP classes use self.__AttributeName (private). When the question says "record" → no underscore, no getters. When it says "OOP class" → double underscore, add getters and setters.

File Processing
Saving records to disk — sequential vs random access
💾 Why binary files? Text files can only store strings, line by line. If you want to store full record objects (with integers, strings, etc.) and be able to jump directly to record number 7 without reading 1–6 first — you need binary files. Python's pickle module handles this.
Sequential File Processing
Records are read or written in order, from start to finish. Like reading a book — you can't skip to chapter 7 without going through 1–6.

✅ Simple to implement
✅ Good for processing all records
❌ Slow to find a specific record
❌ Must read everything before the target
Random Access File Processing
Jump directly to any position in the file using a calculated offset. Like a DVD — skip to any scene instantly.

✅ Fast direct access to any record
✅ No need to read irrelevant records
❌ Needs a hash function to find position
❌ More complex to implement

# ── SEQUENTIAL WRITE ──────────────────────────────────────
import pickle

def WriteRecords(records):
    # "wb" = write binary mode
    with open("teams.dat", "wb") as f:
        for record in records:
            pickle.dump(record, f)    # writes one record at a time, in order

# Usage
teams = [team1, team2, team3]
WriteRecords(teams)
# File now contains all 3 records, written one after another

What is pickle? Python objects in RAM are complex structures — not just text. pickle converts any object into a stream of bytes (pickle.dump = save) and restores it exactly from those bytes later (pickle.load = load). Like freeze-drying food: compress it, store it, restore it perfectly later. 
# ── SEQUENTIAL READ ───────────────────────────────────────
import pickle

def ReadAllRecords():
    records = []
    # "rb" = read binary mode
    with open("teams.dat", "rb") as f:
        while True:
            try:
                record = pickle.load(f)   # read next record
                records.append(record)
            except EOFError:              # end of file reached — stop!
                break
    return records

all_teams = ReadAllRecords()
for t in all_teams:
    print(t.TeamName, t.Points)
Why EOFError? When pickle.load() tries to read past the last record, it raises EOFError (end of file). We catch it to know we've finished reading. This is exception handling being useful for file processing!
Before reading the code — understand the 3-step recipe Every random access write does exactly three things in order: (1) decide which slot to use via a hash function, (2) calculate which byte that slot starts at, (3) jump the cursor there and write. That's it.
The 3-line pattern for WRITE — with "why" for each line
# Code What it does Why you need it
1 position = hash(key) % TABLE_SIZE Converts the key (e.g. team name or ID) into a slot number between 0 and TABLE_SIZE−1 Without this we don't know where in the file to put the record. The hash gives us a consistent, repeatable slot every time for the same key.
2 offset = position × RECORD_SIZE Turns the slot number into a byte address. Slot 3 × 256 bytes = byte 768. The file is just bytes. We need to know which byte our slot starts at. Multiplying by the fixed record size gives us that.
3 f.seek(offset) Moves the file cursor to that exact byte. The next write starts exactly there. Without seek(), Python would write at the current cursor position — which is wherever you left it. seek() lets you jump anywhere instantly. 
# ── RANDOM ACCESS WRITE — full code ──────────────────────
import pickle
 
RECORD_SIZE = 256   # fixed bytes per record — MUST be consistent
TABLE_SIZE  = 100   # number of slots in the file
 
def WriteRecord(record, key):
    # ── LINE 1: hash the key to get a slot ──────────────────
    position = hash(key) % TABLE_SIZE    # e.g. hash("Mercedes") % 100 = 43
 
    # ── LINE 2: convert slot → byte address ─────────────────
    offset = position * RECORD_SIZE        # e.g. 43 × 256 = 11,008
 
    # ── LINE 3: jump to that byte and write ─────────────────
    with open("teams_random.dat", "r+b") as f:
        f.seek(offset)                     # cursor jumps to byte 11,008
        data = pickle.dumps(record)        # convert object → bytes
        data = data.ljust(RECORD_SIZE, b'\x00')  # pad to fixed size
        f.write(data[:RECORD_SIZE])        # write exactly 256 bytes
Why "r+b" not "wb"? "wb" would erase the entire file every time you open it. "r+b" = read+write binary: opens the existing file and lets you overwrite specific bytes without touching the rest. You must create the file first (with all slots pre-filled) before using "r+b".
The 3-line pattern for READ — same logic, reversed
# Code What it does Why it's the same key
1 position = hash(key) % TABLE_SIZE Identical to the write step. Same key → same slot number. This is the magic. hash() always returns the same value for the same input. So the read calculation lands on exactly the same slot as the write did.
2 offset = position × RECORD_SIZE Identical byte address calculation. Same slot → same byte address → cursor lands in exactly the right place.
3 f.read(RECORD_SIZE) Reads exactly RECORD_SIZE bytes from the cursor position. Since every record is the same fixed size, reading exactly RECORD_SIZE bytes always gets exactly one record — no more, no less. 
# ── RANDOM ACCESS READ — full code ───────────────────────
import pickle
 
RECORD_SIZE = 256
TABLE_SIZE  = 100
 
def ReadRecord(key):
    # ── LINE 1: same hash as write ───────────────────────────
    position = hash(key) % TABLE_SIZE    # SAME formula as write → same slot
 
    # ── LINE 2: same offset calculation ─────────────────────
    offset = position * RECORD_SIZE
 
    # ── LINE 3: jump and read ───────────────────────────────
    with open("teams_random.dat", "rb") as f:
        f.seek(offset)                     # jump to exact byte
        data = f.read(RECORD_SIZE)         # read exactly one slot's worth
        data = data.rstrip(b'\x00')       # strip padding bytes
        record = pickle.loads(data)        # bytes → object
    return record
 
result = ReadRecord("Mercedes")
print(result.TeamName)   # output: Mercedes — found in ONE seek, no loop needed
The whole point in one sentence Because we use the same hash formula on read as on write, we always land on the same slot — which means we jump directly to the right bytes without reading anything else in the file.
Understanding random access — properly 🗂️
The stepper above showed you the code. Now let's build the mental model so you can adapt when the exam changes the scenario.
① The problem random access solves

Sequential processing is simple — but it forces you to read every record before the one you want. If you have 10,000 teams and you want team number 9,847, you must read 9,846 records first. That's slow. Random access solves this by jumping directly to the right position in one step — like skipping to a specific page in a book by knowing the page number, rather than flipping through every page from the front.

❌ Sequential — finding "Ferrari"
Read team 1 → not Ferrari
Read team 2 → not Ferrari
Read team 3 → not Ferrari
Read team 4 → Ferrari! ✓
Wasted 3 reads. With 10,000 records, could waste 9,999.
✅ Random access — finding "Ferrari"
hash("Ferrari") % 100 = 37
offset = 37 × 256 = 9,472
seek(9,472) → read → Ferrari! ✓
 
Always 1 seek. Doesn't matter if there are 10 or 10,000 records.
② The numbered shelf analogy

Imagine a warehouse with 100 numbered shelves (0 to 99). Each shelf holds exactly one box. When you store a team's data, you don't just put it on the first free shelf — you let the hash function calculate its shelf number. The rule is simple: same team → always same shelf. So later, when you need to find that team, you hash the name again, get the same shelf number, and walk straight there.

③ What hash() actually does

hash() is a built-in Python function. You don't write it — it already exists. You just call it. The only thing you need to know about it is this one rule:

Same input → always same output
hash("Mercedes") will return the same big integer every single time you call it with "Mercedes"

Exam advice — you don't need to know how hash() works internally. You just need to know: (1) same input → same slot, (2) you use modulo % to keep the result within the valid range, (3) use the exact same formula on both write and read.
④ What seek() actually does — the cursor picture

A file on disk is just a long sequence of bytes, numbered from 0. When Python opens a file, it keeps an invisible cursor — like a blinking text cursor — that marks where the next read or write will happen. Normally the cursor starts at byte 0 and moves forward as you read. f.seek(n) teleports the cursor to byte n instantly, without reading anything in between.

Go back to the very start of the file. Cursor = byte 0.

⑤ What can go wrong — and what the exam expects you to say
⚠ Problem 1: Collision — two records map to the same slot
hash("Mercedes") % 100 = 43 AND hash("Alpine") % 100 = 43 — same slot!
When two different keys produce the same slot number, the second record overwrites the first. This is called a hash collision. The first team's data is gone.
What the exam expects: Name the problem ("hash collision" or "overflow"). State that the solution is an overflow area — a separate section of the file where colliding records are stored. On read, if the slot doesn't match the expected key, check the overflow area.
⚠ Problem 2: Reading an empty slot
You search for "McLaren" but nothing was ever written to that slot.
The slot exists (we pre-filled the file with empty/null records) but it contains no real data. Reading it gives you garbage or a null record.
What the exam expects: After reading a record, check if the key field matches what you searched for. If it doesn't match (or the record is empty/null), the team was not found. Output "Record not found." Don't assume the slot is occupied just because you can seek to it. 
# After reading, ALWAYS verify the record is the right one
record = pickle.loads(data.rstrip(b'\x00'))
if record.TeamName == search_key:
    print("Found:", record.Points)
else:
    print("Record not found")   # empty slot or collision victim
⚠ Problem 3: File doesn't exist yet
"r+b" raises IOError if the file doesn't exist — unlike "wb" which creates it.
Random access requires a pre-existing file with all slots already allocated. You can't use "r+b" on a file that hasn't been created yet.
What the exam expects: Wrap the open() in try/except IOError. Alternatively, initialise the file first by writing TABLE_SIZE empty/null records in sequence before any random access operations begin. 
# Initialise file — write TABLE_SIZE empty slots first
def InitialiseFile():
    empty = b'\x00' * RECORD_SIZE    # one blank slot
    with open("teams_random.dat", "wb") as f:
        for _ in range(TABLE_SIZE):
            f.write(empty)             # write 100 blank slots
# Now the file exists and "r+b" will work
⑥ Real exam question breakdown — what examiners check for
Exam-style question: A file products.dat uses random access. Each record has a fixed size of 512 bytes. Write a procedure WriteProduct(product, key) that writes a product record to the correct position in the file. The hash function is hash(key) % 200. [5 marks] 
# Model answer
import pickle
 
RECORD_SIZE = 512
TABLE_SIZE  = 200
 
def WriteProduct(product, key):
    position = hash(key) % TABLE_SIZE         # [1] hash → slot
    offset   = position * RECORD_SIZE           # [1] slot → byte
    try:
        with open("products.dat", "r+b") as f:  # [1] correct mode
            f.seek(offset)                      # [1] seek to position
            data = pickle.dumps(product)
            data = data.ljust(RECORD_SIZE, b'\x00')
            f.write(data[:RECORD_SIZE])         # [1] write fixed-size bytes
    except IOError:
        print("Error: file could not be opened")
Examiner mark checklist — what they're looking for
[1]
Hash formula applied correctly: hash(key) % TABLE_SIZE — the modulo keeps the slot in range
[1]
Offset calculated: position × RECORD_SIZE — not position alone, not position × some other number
[1]
File opened in "r+b" mode (NOT "wb" which would erase everything, NOT "rb" which is read-only)
[1]
f.seek(offset) called before the write — not after, not forgotten
[1]

Common mistakes that lose marks: Using "wb" instead of "r+b". Writing f.seek(position) instead of f.seek(offset). Forgetting the seek entirely. Not padding to fixed size (means records bleed into each other).

Exception Handling
Catching runtime errors before they crash your program
💥 What is a runtime error (exception)? Some bugs can't be caught before the program runs. Dividing by zero, opening a file that doesn't exist, asking the user for a number and they type "banana" — these all crash your program mid-run. An exception is Python's way of saying "something went wrong right now, at runtime."
🛡️
Without exception handling
Program crashes. User panics.
Your program hits 1/0, throws a ZeroDivisionError, prints a scary traceback, and dies. The user sees gibberish and loses any unsaved work.
✅
With exception handling
Caught. Handled. Continues.
You wrap risky code in try. If it fails, Python jumps to except. You show a nice message, maybe ask again, and keep running. User never sees a crash.

04
Exception Types
The 6 types Cambridge tests — know them cold
ZeroDivisionError
Triggered by: dividing by zero
Mathematically undefined. 10 / 0 → crash. Very common in calculators and grade average programs.
ValueError
Triggered by: wrong value for correct type
Right type, wrong content. int("banana") → crash. The most common one when reading user input.
IOError
Triggered by: file doesn't exist / permission denied
Input/output operation fails. Opening a file that isn't there, writing to a read-only file. Essential for file processing.
EOFError
Triggered by: reading past end of file
Tried to read more data but the file ended. Used in sequential file processing to detect when all records have been read.
ImportError
Triggered by: module not found
import some_module fails because the module isn't installed or doesn't exist. Less common in exams but appears in marking.
KeyboardInterrupt
Triggered by: user presses Ctrl+C
User manually interrupts execution. Useful in long-running programs where you want to handle "please stop" gracefully.

# ── BASIC try / except ─────────────────────────────────────
try:
    number = int(input("Enter a number: "))
    result = 100 / number
    print("Result:", result)
except ZeroDivisionError:
    print("Error: cannot divide by zero!")
except ValueError:
    print("Error: please enter a valid number!")

# User types "0"  → ZeroDivisionError block runs
# User types "abc" → ValueError block runs
# User types "5"  → neither except runs, result = 20.0
# ── EXCEPTION AS e (capture error message) ─────────────────
# "as e" captures the exception object so you can read its message
try:
    result = 10 / 0
except ZeroDivisionError as e:
    print("Caught:", str(e))   # output: Caught: division by zero

# Use this when you want to show the actual error message to the user
# or log it for debugging. Cambridge often asks for this pattern.
# ── FILE PROCESSING with exception handling ─────────────────
try:
    with open("teams.dat", "rb") as f:
        while True:
            try:
                record = pickle.load(f)
                print(record.TeamName)
            except EOFError:
                break                     # end of file → stop reading
except IOError:
    print("Error: file not found or cannot be opened")

# Outer try: catches IOError if file doesn't exist at all
# Inner try: catches EOFError when we've read the last record
# This is the FULL pattern Cambridge expects for file reading
# ── try / except / else / finally ──────────────────────────
try:
    f = open("teams.dat", "rb")
    data = pickle.load(f)
except IOError:
    print("File could not be opened")   # runs ONLY if error
else:
    print("File read successfully!")     # runs ONLY if NO error
    print(data.TeamName)
finally:
    f.close()                            # ALWAYS runs — close the file!
    print("File closed.")
🔎 Why use finally?
finally:
→
This block runs no matter what — whether an exception was raised or not. Perfect for cleanup: closing files, releasing resources, resetting state. If you open a file in try, always close it in finally. (The with statement does this automatically — but Cambridge may ask you to do it manually.)
else:
→
Runs only if the try block completed without any exceptions. Think of it as "the happy path code". Keeps your success logic separate from your error-handling logic — cleaner to read.

E2 Write and read records sequentially — with EOFError handling

A program stores StudentRecord objects in a binary file called students.dat. The program needs two procedures: one to write a list of records to the file, one to read them all back.
  • Write procedure WriteStudents(records) that takes a list of StudentRecord objects and writes them sequentially to students.dat. [3]
  • Write procedure ReadStudents() that reads all records from students.dat and prints each student's name and grade. Use appropriate exception handling to detect end of file AND handle a missing file. [5]
Cambridge expects: import pickle, correct file modes ("wb" / "rb"), EOFError to stop reading, and IOError for missing file.
Write: open("students.dat","wb"), loop through records, pickle.dump(record, f). Read: outer try/except IOError wraps the open. Inner while True: try: pickle.load / except EOFError: break. 
import pickle

# ── WRITE ──────────────────────────────────────────────────
def WriteStudents(records):
    with open("students.dat", "wb") as f:
        for record in records:
            pickle.dump(record, f)

# Mark scheme WriteStudents (3 marks):
# [1] opens file in binary write mode "wb"
# [1] loops through records list
# [1] pickle.dump(record, f) called correctly


# ── READ ───────────────────────────────────────────────────
def ReadStudents():
    try:
        with open("students.dat", "rb") as f:
            while True:
                try:
                    record = pickle.load(f)
                    print(record.Name, record.Grade)
                except EOFError:
                    break           # no more records — exit loop
    except IOError:
        print("Error: could not open students.dat")

E4 Full program — records + file I/O + exception handling

A program manages a product inventory. Each product has: ProductID (STRING), Name (STRING), Price (REAL), Quantity (INTEGER). Products are stored in a binary file inventory.dat.
  • Declare the record type ProductRecord. [2]
  • Write procedure SaveProduct(product) that appends one product to inventory.dat using binary write (append mode). Handle the case where the file cannot be opened. [3]
  • Write function LoadAllProducts() that reads and returns a list of all products. Use exception handling for end of file and for a missing file. If the file is missing, return an empty list. [4]
  • Write a main program section that: loads all products, asks the user for a product name to search, and prints the price and quantity if found, or "Not found" if not. Wrap the user input in exception handling. [3]
Append mode in Python: "ab" = append binary. Opens the file and adds to the end without overwriting existing data.
SaveProduct: open with "ab" mode, pickle.dump, wrap in try/except IOError. LoadAllProducts: outer try/except IOError returns []. Inner while True loop with EOFError break. Return the list. Main: call LoadAllProducts(), get user input, loop through list comparing .Name, print result. 
import pickle

# ── PART 1: Record type ────────────────────────────────────
class ProductRecord:
    def __init__(self, product_id, name, price, quantity):
        self.ProductID = product_id   # STRING
        self.Name      = name         # STRING
        self.Price     = price        # REAL
        self.Quantity  = quantity     # INTEGER


# ── PART 2: Save (append) ──────────────────────────────────
def SaveProduct(product):
    try:
        with open("inventory.dat", "ab") as f:   # "ab" = append binary
            pickle.dump(product, f)
    except IOError:
        print("Error: could not save product to file")


# ── PART 3: Load all ───────────────────────────────────────
def LoadAllProducts():
    products = []
    try:
        with open("inventory.dat", "rb") as f:
            while True:
                try:
                    product = pickle.load(f)
                    products.append(product)
                except EOFError:
                    break
    except IOError:
        print("No inventory file found — starting fresh")
    return products   # returns [] if file not found


# ── PART 4: Main program ───────────────────────────────────
products = LoadAllProducts()

try:
    search = input("Enter product name to search: ")
    found = False
    for p in products:
        if p.Name.lower() == search.lower():
            print(f"Price: {p.Price}  |  Quantity: {p.Quantity}")
            found = True
            break
    if not found:
        print("Not found")
except Exception as e:
    print("Unexpected error:", str(e))
