01
What even is OOP?
The big idea — and why it exists
🎮 Real life analogy — a video game character Think of a video game. Your character has:
• Data — name, health, position, level
• Actions — move, attack, jump, heal

In normal (procedural) programming, you'd have 10 separate variables for the data and 10 separate functions for the actions, all floating around separately. Messy.

OOP says: pack the data AND the actions together into one neat box — called an object. Your character is ONE thing that knows its own health AND knows how to heal itself.
🏭
Class
The factory blueprint
A class is like a blueprint or mould. It describes WHAT a thing looks like and WHAT it can do — but it's not the thing itself. A car blueprint isn't a car.
🚗
Object
The actual car
An object is a real instance made from the blueprint. You can make 1000 cars from the same blueprint — each car is a separate object with its own colour, mileage, etc.
📦
Attributes
The stored data
Attributes are the variables inside the object. For a car: colour, speed, fuel level, registration. Each object has its own copy of these values.
🔧
Methods
The actions
Methods are the functions inside the class that work on the attributes. For a car: accelerate(), brake(), refuel(). They're the verbs — things the object CAN DO.
Key Terms — know these cold 🧊
TermWhat it meansReal-world example
Class A template/blueprint for creating objects. Defines attributes and methods. The blueprint for a house
Object A specific instance created from a class. Has real values in its attributes. The actual house at 42 Oak Street
Attribute A variable that belongs to a class/object. Stores the object's state/data. Number of bedrooms, colour of walls
Method A function defined inside a class that operates on the object's attributes. openDoor(), lockWindow()
Encapsulation Bundling data + methods together AND keeping data private. The data is hidden. The house's wiring is hidden in walls — you use the light switch, not the wires
Constructor A special method (__init__ in Python) that runs automatically when an object is created. The building process — sets everything up
Instantiation The act of creating an object from a class. You "instantiate" a class. Actually building the house from the blueprint
Getter A method that reads (returns) a private attribute's value. An estate agent checking the house's price for you
Setter A method that updates a private attribute's value (with validation). Officially changing the registered address — goes through a process
01b
Why do we even need OOP?
Without it, your code becomes spaghetti. With it — clean, reusable, powerful.
The problem — imagine a school app 🏫
You need to store data for 500 students. Each student has: name, age, grade, score. And you need to calculate their grade. Without OOP, here's what happens...
❌ Without OOP (procedural mess)
# Student 1
student1_name  = "Ali"
student1_age   = 17
student1_score = 85

# Student 2
student2_name  = "Sara"
student2_age   = 16
student2_score = 92

# Student 3...
student3_name  = "Rami"
# ... ×500 students
# COPY-PASTE NIGHTMARE 😭

# Calculate grade for student1
if student1_score >= 90:
    student1_grade = "A"
# Same code again for student2
# Same code again for student3
# × 500 times...
✅ With OOP (clean & powerful)
# Define the blueprint ONCE
class Student:
    def __init__(self, name, age, score):
        self.__Name  = name
        self.__Age   = age
        self.__Score = score

    def CalcGrade(self):
        if self.__Score >= 90:
            return "A"
        # logic written ONCE

# Now create 500 students
# in 3 lines each — or a loop!
s1 = Student("Ali",  17, 85)
s2 = Student("Sara", 16, 92)
s3 = Student("Rami", 15, 78)
# Grade logic reused automatically ✅
💡 The key insight Without OOP: you write the same code 500 times. Change one thing? Update 500 places. Find a bug? Check 500 places. That's a nightmare.
With OOP: you write the blueprint once. Every student object automatically gets all the attributes and methods. Change the blueprint → ALL students update instantly.
The Student class → three student objects 🎓
One class. Three real objects. Each has its own values — but the same structure.
🧠 What just happened? We wrote the Student class once. Then we created three objects (s1, s2, s3) from it. Each object is called an instance. They all have the same structure (Name, Age, Score, CalcGrade) but each one has its own personal values. Same mould — different chocolate bars.
01c
What is an Instance? 🍪
The most confusing word in OOP — finally made simple
🍪 Instance = one specific cookie made from the cutter The class is the cookie cutter. The instance is the actual cookie. The cutter defines the shape — but every cookie you make is its own thing. You can eat one cookie without destroying the others. You can change Ali's score without changing Sara's score. Each instance is independent.

Object = Instance — these two words mean exactly the same thing. Cambridge uses both. Don't get confused.
01d
What is __init__ and why is it ALWAYS there?
The constructor — the setup crew that runs before anything else
🏗️ The hotel room analogy When a hotel guest arrives, a setup crew goes in first: makes the bed, puts fresh towels, sets the temperature. They run AUTOMATICALLY before the guest enters. The guest doesn't call them — it just happens.

__init__ is that setup crew. The moment you create an object, __init__ runs automatically and sets up all the attributes. You never call it yourself — Python calls it for you.
01e
What is self — really?
The most confusing keyword for beginners — zero mystery after this
🪞 self = "me, myself, this specific object" Imagine you have three students: Ali, Sara, Rami. They all have a name. When Ali says "my name", he means HIS name, not Sara's. When Sara says "my name", she means HER name.

self is Python's version of "my" / "mine". When code runs on Ali's object, self.__Name means Ali's name. When the same code runs on Sara's object, self.__Name means Sara's name. Same code — different object.
🔎 Why do we write self in every method?
def GetName(self):
→
self must be the first parameter of every method. Python automatically passes the object in there when you call s1.GetName(). You don't type it yourself when calling — Python does it.
self.__Name
→
Means "the __Name attribute of THIS object". Without self, Python would look for a regular variable called __Name and crash. self tells Python: "look inside THIS object's memory."
Why not just call it "this"?
→
Python uses self by convention. Other languages (Java, C++) use this. Same idea. You could technically call it anything — but calling it self is the rule everyone follows.
01f
Why Getters & Setters? 🔐
Why not just read and change attributes directly?
🏦 The bank account analogy Your bank stores your balance. Can you just walk in and change your balance to £1,000,000 yourself? No. You go through official channels: you deposit money, you withdraw money. There are RULES. The bank validates everything first.

That's exactly what setters do. The balance (attribute) is private. The deposit/withdraw methods (setters) are the official channels that check if the operation is valid before changing anything.
👁️ Getter — why use it? The attribute is private. You can't read it directly from outside. A getter is a safe window — it lets you see the value without being able to mess with it. It just returns the value, nothing else.

def GetScore(self):
    return self.__Score
✍️ Setter — why use it? You want to change a value — but safely. The setter checks if the new value is valid before updating. Without it, anyone could set score to -999 or 99999. With it, you control what's allowed.

def SetScore(self, val):
    if 0 <= val <= 100:
        self.__Score = val

Anatomy of a Class
Every class has exactly three parts — header, attributes, methods
The UML class diagram — how Cambridge draws it 📐
Cambridge always shows the class as a 3-section table. You need to read AND write these.
Car
Attributes (data)
−ID: STRING
−Colour: STRING
−Speed: INTEGER
−MaxSpeed: INTEGER
Methods (actions)
+Constructor()
+GetSpeed()
+GetID()
+SetColour()
+Accelerate()
− means PRIVATE The attribute is hidden from the outside world. Code outside the class CANNOT directly access or change it. This is encapsulation.
+ means PUBLIC The method is accessible from outside. Anyone can call it. Methods are almost always public — they are the "interface" you interact with.
🎯 Cambridge exam tip The top box = class name. Middle box = attributes (usually all private with −). Bottom box = methods (usually all public with +). Always write the Constructor() first in the methods.
03
Writing a Class in Python
Every single line explained — nothing skipped
Step 1 — Declaring the class and constructor 🏗️
class Car:
    # ── CONSTRUCTOR ──────────────────────────────
    # This runs AUTOMATICALLY when you create a Car object.
    # "self" always refers to THIS specific object being created.
    def __init__(self, car_id, colour, max_speed):

        # __ before the name = PRIVATE attribute
        # Nobody outside the class can touch these directly
        self.__ID       = car_id
        self.__Colour   = colour
        self.__Speed    = 0        # car starts at rest
        self.__MaxSpeed = max_speed
🔎 What does self mean?
class Car:
→
Creates a blueprint called Car. Not a real car yet — just the template.
def __init__(self, ...)
→
The constructor. self is Python's way of saying "this particular object I'm creating right now". Python passes it automatically — you never type it when calling.
self.__ID = car_id
→
Stores the value of parameter car_id into this object's private attribute __ID. The __ makes it private — Python name-mangles it so outside code can't easily access it.
self.__Speed = 0
→
Some attributes don't come from parameters — they have a fixed starting value. Speed starts at 0 (car isn't moving yet). This is set up in the constructor automatically every time.
Step 2 — Getters (read private data) 👁️
Because attributes are private, you can't do my_car.__Speed from outside. Getters solve this.
    # ── GETTER METHODS ───────────────────────────
    # Named GetXxx — return the value, that's it.

    def GetID(self):
        return self.__ID

    def GetColour(self):
        return self.__Colour

    def GetSpeed(self):
        return self.__Speed

    def GetMaxSpeed(self):
        return self.__MaxSpeed
Why do getters exist? If __Speed is private, you literally cannot do my_car.__Speed from outside the class. The getter GetSpeed() is the only official way to read it. This is intentional — it keeps you in control. You decide what data people can see, and how.
Step 3 — Setters (change private data with validation) 🔒
Setters can reject bad values — a raw assignment outside the class can't.
    # ── SETTER METHODS ───────────────────────────
    # Named SetXxx — validate first, then update.

    def SetColour(self, new_colour):
        if new_colour != "":              # can't set an empty string
            self.__Colour = new_colour

    def SetSpeed(self, new_speed):
        if 0 <= new_speed <= self.__MaxSpeed:   # must be valid range
            self.__Speed = new_speed
Why not just let people change the attribute directly? If speed was public, someone could do my_car.Speed = 99999 — breaking the car logic. With a setter, you check: "Is this a valid speed? Is it ≤ MaxSpeed?" Only then do you allow the change. Validation = control.
Step 4 — Other methods (the actual behaviour) ⚡
    # ── OTHER METHODS ────────────────────────────

    def Accelerate(self, amount):
        new_speed = self.__Speed + amount
        if new_speed > self.__MaxSpeed:
            self.__Speed = self.__MaxSpeed   # cap at max
        else:
            self.__Speed = new_speed

    def DisplayInfo(self):
        print(f"Car: {self.__ID} | Colour: {self.__Colour}")
        print(f"Speed: {self.__Speed} / {self.__MaxSpeed}")

Encapsulation
Hiding the wiring — the most important OOP concept
🔒 The capsule analogy Think of a medicine capsule. The medicine is INSIDE — protected, measured, safe. You don't interact with the powder directly. You just swallow the capsule. That's encapsulation: the data (medicine) is bundled inside the class (capsule), and you only interact through the controlled interface (swallowing).
❌
Without encapsulation
Direct access (BAD)
car1.Speed = 99999

Anyone can set speed to anything. No validation. Can corrupt the object's state. Impossible to track where the data changed.
✅
With encapsulation
Through setter (GOOD)
car1.SetSpeed(50)

Goes through the setter. Setter validates: is 50 ≤ MaxSpeed? Yes → accepted. Always safe. You control exactly how data changes. 
# WHY __ MAKES ATTRIBUTES PRIVATE IN PYTHON
# Python doesn't have "true" private — __ triggers name mangling.
# The real name becomes _ClassName__AttributeName

car1 = Car("ABC", "red", 200)

# This WON'T WORK as expected (generates AttributeError or different var):
# print(car1.__Speed)  → AttributeError

# This is the CORRECT way (through the getter):
print(car1.GetSpeed())    # ✅ works — output: 0

# Cambridge exam rule: always declare private attributes with __ prefix
# Always access them via getters, always change via setters.
📌 Cambridge mark scheme — what they want to see When asked to "declare private attributes": use self.__attributeName in __init__ When asked to "write a getter for X": a method that returns self.__X When asked to "write a setter for X": a method that optionally validates, then sets self.__X All attributes must be private in Cambridge OOP questions unless told otherwise.

Inheritance
Child class = parent class + extra stuff
🧬 The DNA analogy You inherited your parents' eye colour, height, nose shape — but you also have your own personality and skills they don't have. That's inheritance in OOP: a subclass (child) automatically gets all the attributes and methods from the superclass (parent), and can add its own on top.
The inheritance hierarchy 🌳
Vehicle (Superclass / Parent)

− ID : STRING

− MaxSpeed : INTEGER

− CurrentSpeed : INTEGER

+ Constructor()

+ GetID()

+ GetSpeed()

+ Accelerate()

↓   inherits   ↓
Car (Subclass / Child)

− Doors : INTEGER

+ all Vehicle attributes

+ Constructor()

+ GetDoors()

+ all Vehicle methods
Helicopter (Subclass)

− VerticalPos : INTEGER

− MaxHeight : INTEGER

+ all Vehicle attributes

+ Constructor()

+ GetVerticalPos()

+ Accelerate() ← overrides!

Both Car and Helicopter can use GetID(), GetSpeed(), etc. — without rewriting them
Writing inheritance in Python 🐍
# ── SUPERCLASS (Parent) ─────────────────────────────────────
class Vehicle:
    def __init__(self, vehicle_id, max_speed):
        self.__ID           = vehicle_id
        self.__MaxSpeed    = max_speed
        self.__CurrentSpeed = 0

    def GetID(self):         return self.__ID
    def GetMaxSpeed(self):  return self.__MaxSpeed
    def GetSpeed(self):     return self.__CurrentSpeed

    def Accelerate(self, amount):
        new_speed = self.__CurrentSpeed + amount
        if new_speed > self.__MaxSpeed:
            self.__CurrentSpeed = self.__MaxSpeed
        else:
            self.__CurrentSpeed = new_speed


# ── SUBCLASS (Child) ────────────────────────────────────────
# class ChildName(ParentName):  ← this is how you inherit
class Car(Vehicle):

    def __init__(self, vehicle_id, max_speed, doors):
        # Call parent's constructor first with super().__init__()
        # This sets up all Vehicle attributes automatically
        super().__init__(vehicle_id, max_speed)

        # Then add Car's own new attribute
        self.__Doors = doors

    def GetDoors(self):
        return self.__Doors


# ── ANOTHER SUBCLASS ────────────────────────────────────────
class Helicopter(Vehicle):

    def __init__(self, vehicle_id, max_speed, max_height):
        super().__init__(vehicle_id, max_speed)   # parent setup
        self.__VerticalPos = 0
        self.__MaxHeight  = max_height

    def GetVerticalPos(self):
        return self.__VerticalPos


# ── USING THEM ──────────────────────────────────────────────
my_car = Car("CAR001", 200, 4)
heli   = Helicopter("HEL001", 350, 1000)

print(my_car.GetID())        # from Vehicle — output: CAR001
print(my_car.GetDoors())    # from Car — output: 4
my_car.Accelerate(60)        # from Vehicle — works on Car!
print(my_car.GetSpeed())    # from Vehicle — output: 60
🔎 What does super().__init__() actually do?
super()
→
Refers to the parent class (Vehicle in this case). "Hey parent, I need you."
.__init__(id, speed)
→
Calls the parent's constructor. This creates all the Vehicle attributes (ID, MaxSpeed, CurrentSpeed) inside our Car object. Without this, the Car wouldn't have any Vehicle attributes!
Why not just copy the code?
→
Because if Vehicle changes, Car automatically gets the update. Write once, reuse everywhere. That's the whole point of inheritance — no copy-paste duplication.
07
Polymorphism
Same method name → different behaviour depending on the class
🎭 Polymorphism = "many forms" Imagine you text your friends saying "make dinner". Your Italian friend makes pasta. Your Tunisian friend makes couscous. Your Korean friend makes bibimbap. Same message — completely different results. That's polymorphism: the same method name does different things depending on which class's object you call it on.
Step through the concept
# Step 1: Parent class defines the method
class Vehicle:
    def Move(self):
        print("Vehicle is moving")    # generic
The parent defines Move() in a basic way. All vehicles can move. But HOW they move is different.
# Step 2: Car OVERRIDES the method
class Car(Vehicle):
    def Move(self):              # same name — overrides parent!
        super().Move()           # optional: call parent version first
        print("Car driving on the road at", self.GetSpeed(), "km/h")
Car has its own Move() that replaces the parent's. When Python sees my_car.Move(), it uses Car's version, not Vehicle's.
# Step 3: Helicopter also overrides — differently!
class Helicopter(Vehicle):
    def Move(self):
        super().Move()
        print("Helicopter flying at altitude", self.GetVerticalPos(), "m")
Helicopter has its OWN version of Move(). Same method name, completely different behaviour.
# Step 4: Calling Move() on both — different outputs!
my_car = Car("CAR001", 200, 4)
heli   = Helicopter("HEL001", 350, 1000)

my_car.Move()
# output: Vehicle is moving
#         Car driving on the road at 0 km/h

heli.Move()
# output: Vehicle is moving
#         Helicopter flying at altitude 0 m

# SAME method name (.Move()) → DIFFERENT behaviour. That's polymorphism!
This is what Cambridge exams test: Does the child class override a method? Does it call super().method() first? Does it add its own behaviour?

Containment (Aggregation)
An object storing other objects inside it
📚 The library analogy A library CONTAINS books. The library is one object. Each book is another object. The library stores a list of book objects inside it. This is containment (also called aggregation — objects holding other objects). 
# ── CONTAINED class (the book) ──────────────────────────────
class Book:
    def __init__(self, title, author):
        self.__Title  = title
        self.__Author = author

    def GetTitle(self):   return self.__Title
    def GetAuthor(self):  return self.__Author

    def DisplayDetails(self):
        print(f"'{self.__Title}' by {self.__Author}")


# ── CONTAINER class (the library) ──────────────────────────
class Library:
    def __init__(self):
        self.__Books = []    # ← containment happens here (list of Book objects)

    def AddBook(self, book):          # takes a Book object as parameter
        self.__Books.append(book)

    def DisplayAllBooks(self):
        for book in self.__Books:
            book.DisplayDetails()   # calling Book's method from Library!


# ── CREATING THE RELATIONSHIP ───────────────────────────────
book1 = Book("Python Crash Course", "Eric Matthes")
book2 = Book("Clean Code", "Robert Martin")

my_library = Library()
my_library.AddBook(book1)   # Library now contains book1
my_library.AddBook(book2)   # Library now contains book2

my_library.DisplayAllBooks()
# output:
# 'Python Crash Course' by Eric Matthes
# 'Clean Code' by Robert Martin

Vehicle constructor: 3 params (ID, MaxSpeed, IncreaseAmount), set CurrentSpeed=0 and HorizontalPosition=0 manually. IncreaseSpeed: new_speed = current + increase_amount. If > MaxSpeed, cap. # ══════════════════════════════════════════════════════════════
# VEHICLE CLASS
# ══════════════════════════════════════════════════════════════
class Vehicle:

    def __init__(self, vehicle_id, max_speed, increase_amount):
        # ID : STRING
        # MaxSpeed : INTEGER
        # CurrentSpeed : INTEGER
        # IncreaseAmount : INTEGER
        # HorizontalPosition : INTEGER
        self.__ID                 = vehicle_id
        self.__MaxSpeed          = max_speed
        self.__IncreaseAmount   = increase_amount
        self.__CurrentSpeed     = 0
        self.__HorizontalPosition = 0

    # ── GETTERS ─────────────────────────────────────────────────
    def GetCurrentSpeed(self):
        return self.__CurrentSpeed

    def GetIncreaseAmount(self):
        return self.__IncreaseAmount

    def GetMaxSpeed(self):
        return self.__MaxSpeed

    def GetHorizontalPosition(self):
        return self.__HorizontalPosition

    # ── SETTERS ─────────────────────────────────────────────────
    def SetCurrentSpeed(self, val):
        self.__CurrentSpeed = val

    def SetHorizontalPosition(self, val):
        self.__HorizontalPosition = val

    # ── INCREASE SPEED ──────────────────────────────────────────
    def IncreaseSpeed(self):
        new_speed = self.__CurrentSpeed + self.__IncreaseAmount
        if new_speed > self.__MaxSpeed:
            new_speed = self.__MaxSpeed
        self.__CurrentSpeed = new_speed
        self.__HorizontalPosition += self.__CurrentSpeed


# ══════════════════════════════════════════════════════════════
# HELICOPTER CLASS (inherits Vehicle)
# ══════════════════════════════════════════════════════════════
class Helicopter(Vehicle):

    def __init__(self, vehicle_id, max_speed, increase_amount,
                 vertical_change, max_height):
        # VerticalPosition : INTEGER
        # VerticalChange : INTEGER
        # MaxHeight : INTEGER
        super().__init__(vehicle_id, max_speed, increase_amount)
        self.__VerticalPosition = 0
        self.__VerticalChange  = vertical_change
        self.__MaxHeight       = max_height

    def GetVerticalPosition(self):
        return self.__VerticalPosition

    def IncreaseSpeed(self):       # overrides Vehicle's version
        # 1. Update vertical position
        new_vert = self.__VerticalPosition + self.__VerticalChange
        if new_vert > self.__MaxHeight:
            new_vert = self.__MaxHeight
        self.__VerticalPosition = new_vert

        # 2. Update speed (cap at MaxSpeed)
        new_speed = self.GetCurrentSpeed() + self.GetIncreaseAmount()
        if new_speed > self.GetMaxSpeed():
            new_speed = self.GetMaxSpeed()
        self.SetCurrentSpeed(new_speed)

        # 3. Update horizontal position
        self.SetHorizontalPosition(
            self.GetHorizontalPosition() + self.GetCurrentSpeed()
        )
